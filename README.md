# Hadoop MapReduce Simulation (Docker + Python Sockets)

This project simulates a Hadoop-style MapReduce workflow using:
- 1 master container
- 2 worker containers
- Raw TCP sockets in Python (no real Hadoop, no Flask, no REST)

Everything runs with one command after setup.

## 1. Prerequisites

Make sure these are installed:
- Docker Desktop (running)
- Docker Compose (comes with Docker Desktop in most installs)

Quick checks:
- Open PowerShell in this project folder
- Run: docker --version
- Run: docker compose version

If both commands show versions, you are ready.

## 2. Project Files and Roles

- docker-compose.yml
  - Orchestrates master, worker1, worker2 on one shared network.
- Dockerfile
  - Builds a Python 3.10 slim image used by all services.
- master.py
  - Creates data.csv if missing, splits data into 2 chunks, sends map tasks, receives map results, reduces, prints final output, writes output.txt.
- worker.py
  - Listens on TCP 5000, receives chunk, performs word-count map, sends JSON result back.
- data.csv
  - Input data (10,000 lines). Auto-generated by master if missing.
- output.txt
  - Final reduced word count output.

## 3. One-Command Run

From this folder, run:

  docker-compose up --build

What this does:
1. Builds images from Dockerfile.
2. Creates a project network so containers can resolve each other by container name.
3. Creates and starts containers:
   - master
   - worker1
   - worker2
4. Streams logs from all containers.

Keep this running while you observe logs.

## 4. How Images and Containers Are Made

When you run docker-compose up --build:

1. Image build phase
- Docker reads Dockerfile.
- Base image: python:3.10-slim.
- requirements.txt is copied and installed.
- Python files are copied into /app.
- Compose builds one image per service name (same Dockerfile context), commonly named like:
  - hadoop_simulation-master
  - hadoop_simulation-worker1
  - hadoop_simulation-worker2

2. Network phase
- Compose creates a bridge network:
  - hadoop_simulation_hadoop_net
- Containers communicate by service/container names over this network.

3. Container phase
- worker1 starts worker.py and listens on port 5000 internally.
- worker2 starts worker.py and listens on port 5000 internally.
- master starts master.py.

4. Job execution phase
- master ensures data.csv exists (creates 10,000 lines if missing).
- master splits file into two equal chunks.
- master sends chunk1 to worker1:5000 and chunk2 to worker2:5000.
- workers compute local word counts and send JSON back to master:5000.
- master reduces both dicts and prints top 10 words.
- master writes full output to output.txt.

## 5. What You Should See in Logs

Expected key lines include:
- Master: Splitting file...
- Worker1: Received 5000 lines
- Worker2: Received 5000 lines
- Worker1: Sending result complete
- Worker2: Sending result complete
- FINAL REDUCE OUTPUT:
- Master: Job finished successfully

If you see FINAL REDUCE OUTPUT with top words and counts, simulation is working.

## 6. Step-by-Step Testing Plan

Use this exact test flow.

Test A: Fresh run with existing data
1. Run:
   docker-compose up --build
2. Confirm both workers start and wait for map task.
3. Confirm master sends chunks to both workers.
4. Confirm FINAL REDUCE OUTPUT appears.
5. Open a second terminal and verify output file:
   Get-Content .\output.txt -First 10

Pass condition:
- FINAL REDUCE OUTPUT appears in logs.
- output.txt contains word,count rows.

Test B: Force data regeneration path
1. Stop current run with Ctrl+C in the running terminal.
2. Remove data file:
   Remove-Item .\data.csv
3. Run again:
   docker-compose up --build
4. Confirm master logs:
   - not found or empty. Generating 10000 lines...
5. Verify line count:
   (Get-Content .\data.csv | Measure-Object -Line).Lines

Pass condition:
- data.csv is recreated.
- Line count is 10000.
- FINAL REDUCE OUTPUT still appears.

Test C: Verify Docker resources
1. List images:
   docker images
2. List running containers:
   docker ps
3. List all containers (including exited):
   docker ps -a
4. List networks:
   docker network ls
5. Inspect compose network:
   docker network inspect hadoop_simulation_hadoop_net

Pass condition:
- Images for master/worker services exist.
- Containers master, worker1, worker2 are created.
- Compose network exists and containers are attached.

## 7. Useful Commands During Demo

Start and build:
- docker-compose up --build

Run in background:
- docker-compose up --build -d

Follow logs when running detached:
- docker-compose logs -f

Show only master logs:
- docker-compose logs -f master

Show only worker1 logs:
- docker-compose logs -f worker1

Show only worker2 logs:
- docker-compose logs -f worker2

Stop and remove containers/network:
- docker-compose down

Stop and also remove volumes:
- docker-compose down -v

## 8. Quick Troubleshooting

Connection refused:
- Usually startup timing.
- This project already retries socket connections in master and workers.
- Re-run: docker-compose up --build

Port issues:
- Internal container networking is used.
- No host port mapping is required.
- Make sure no custom edits changed port 5000 in code.

File not found:
- Ensure you run commands from this project folder.
- data.csv is auto-created by master if missing.

No logs visible from workers:
- The compose commands use unbuffered python mode, so logs should stream immediately.

## 9. Clean Shutdown

After you finish:
- Press Ctrl+C in compose terminal
- Then run:
  docker-compose down

This stops and removes containers and network cleanly.
