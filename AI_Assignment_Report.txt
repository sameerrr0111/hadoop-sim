B. ARCHITECTURE EXPLANATION
==============================

1) Project Goal in Simple Words
--------------------------------
In our group project, we wanted to show the core idea of Hadoop MapReduce without installing real Hadoop.
So we built a mini-distributed system using Docker and Python sockets.

Our objective was:
- split a big text file into parts
- process parts in parallel using workers
- combine partial results into one final result

This is the same high-level idea used in real big data processing systems.


2) Big Picture Architecture (Who Does What)
--------------------------------------------
Our system has 3 containers, each with one responsibility:

- master container
  - Acts like JobTracker + Reducer in simplified form
  - Reads input file
  - Splits data
  - Sends tasks to workers
  - Collects worker outputs
  - Performs final reduce
  - Prints top 10 words and writes output.txt

- worker1 container
  - Acts as Mapper node
  - Receives one chunk from master
  - Performs word count map
  - Sends partial dictionary back

- worker2 container
  - Same as worker1, processing the second chunk in parallel

Why this design is beginner-friendly:
- each container has one clear role
- behavior can be tracked from logs line-by-line
- no complex frameworks are hidden in background


3) Why We Used Docker Compose
-----------------------------
As students, we used Docker Compose so that setup is one command and reproducible on any laptop.

Compose gives us:
- automatic creation of services (master, worker1, worker2)
- one shared private network
- stable service names for communication
- same environment for all teammates

When we run docker-compose up --build, Docker:
1. builds images from Dockerfile
2. creates network hadoop_simulation_hadoop_net
3. starts all containers
4. streams logs from all services


4) Networking and Communication Details
---------------------------------------
Communication type:
- raw TCP sockets

Payload type:
- JSON messages

Port used:
- 5000 (inside containers)

How addressing works:
- master sends to worker1:5000 and worker2:5000
- workers send results back to master:5000

Important beginner note:
- We do not need to expose port 5000 to host machine for this assignment.
- Containers talk to each other internally through Docker network.

Message style used:
- Task message from master: includes type = MAP_TASK and chunk text
- Result message from worker: includes type = MAP_RESULT, worker name, and word_count dictionary


5) Data Layer and File Handling
-------------------------------
Input data file:
- data.csv

If data.csv does not exist:
- master creates it automatically with 10,000 lines of random English words

Why we added auto-generation:
- no manual preprocessing needed
- demo always works even on first run
- helps beginners run project instantly

File split strategy:
- master reads all lines
- computes split index = total_lines // 2
- chunk1 = first half
- chunk2 = second half

In our test run:
- total lines = 10000
- chunk1 lines = 5000
- chunk2 lines = 5000


6) Detailed Map Phase (Step-by-Step)
------------------------------------
Map phase is done in parallel by worker1 and worker2.

Step flow:
1. master serializes chunk into JSON
2. master opens TCP connection to worker
3. master sends full chunk payload
4. worker receives payload and parses JSON
5. worker tokenizes text into words
6. worker normalizes each token (lowercase and removes punctuation)
7. worker counts each word in dictionary format
8. worker sends partial word_count back to master as JSON

Output format of each worker:
- dictionary like {"word": 120, "data": 90, ...}

Why map phase is important:
- heavy counting work is distributed
- each worker handles smaller data block
- this mimics data-parallel processing in Hadoop


7) Detailed Reduce Phase (Step-by-Step)
---------------------------------------
Reduce is handled by master.

Step flow:
1. master opens server socket on port 5000
2. waits until both worker results are received
3. for each dictionary, loops over all keys
4. adds counts into final_counts dictionary
5. sorts words by frequency descending
6. prints FINAL REDUCE OUTPUT top 10 words
7. writes full sorted output to output.txt

Example reduce logic in plain language:
- if worker1 says map=50 and worker2 says map=40
- final map count becomes 90

This is the same concept as Hadoop reduce: combine intermediate key-value pairs into final aggregated output.


8) Sequence of Events During One Complete Run
----------------------------------------------
This is the exact order beginners can imagine:
1. all containers start
2. workers start listening for tasks
3. master starts
4. master checks/creates data.csv
5. master splits file
6. master sends chunk1 to worker1
7. master sends chunk2 to worker2
8. workers compute local counts
9. workers send results to master
10. master receives both results
11. master reduces and prints top 10
12. master writes output.txt
13. job ends successfully


9) Reliability Features We Added
--------------------------------
Distributed systems can fail due to timing, so we added simple reliability controls:

- Retry mechanism for socket connections
  - If connection fails on first try, retry multiple times with delay

- Defensive JSON parsing
  - Invalid/empty payload handling in worker and master

- Clear logs for each major action
  - easy to identify where failure occurred

- Reusable address option on sockets
  - helps restart containers without address reuse issues

These features helped us avoid random startup failures and made demos stable.


10) Why This Architecture Matches Hadoop Conceptually
-----------------------------------------------------
Even though this is not real Hadoop, concept mapping is very close:

- Input split
  - Hadoop InputSplit -> our chunk1/chunk2

- Mapper stage
  - Hadoop Mapper tasks -> our worker1/worker2 map count

- Shuffle/transfer
  - Hadoop framework moves intermediate data -> our JSON over TCP back to master

- Reducer stage
  - Hadoop Reducer -> our master reduce merge logic

- Final output
  - Hadoop output files -> our output.txt

So this project demonstrates the distributed processing idea correctly at academic level.


11) Beginner Analogy (Easy Memory Trick)
----------------------------------------
Think of this like a class assignment checking words in notebooks:

- Teacher = master
- Student A = worker1
- Student B = worker2
- Big notebook = data.csv

Teacher tears notebook into 2 equal sets of pages.
Each student counts words on their pages.
Students return counts.
Teacher adds both counts and announces top words.

That is exactly what our MapReduce simulation does.

13) Final Architecture Summary
------------------------------
To summarize in one line:
Our architecture is a 1-master + 2-worker distributed word-count pipeline using Docker network and raw TCP sockets, where map work is parallelized and reduce is centralized.

Because of clear role separation, deterministic flow, and detailed logs, even beginners can understand and verify each phase.
